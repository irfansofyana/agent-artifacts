Headroom + LiteLLM on AWS EKS

A practical integration path for adding Headroom prompt compression to a mirrored LiteLLM Docker image running as a single replica on EKS.

Published artifactUpdated 2026-06-30

Executive Summary

Recommended path: install headroom-ai into the mirrored LiteLLM Docker image and enable Headroom’s LiteLLM callback in the existing LiteLLM config.yaml.

This is the least invasive option for a single-replica LiteLLM proxy on EKS. It keeps the existing LiteLLM service shape, works across LiteLLM-supported providers, and avoids adding another proxy hop. Headroom also offers ASGI middleware and a standalone proxy, but those are better as a second phase if you specifically need response headers or proxy-level features.

Sources: - Headroom LiteLLM integration: https://github.com/headroomlabs-ai/headroom/blob/main/docs/content/docs/litellm.mdx - Headroom PyPI package: https://pypi.org/project/headroom-ai/ - Headroom configuration: https://github.com/headroomlabs-ai/headroom/blob/main/docs/content/docs/configuration.mdx - LiteLLM request hook docs: https://docs.litellm.ai/docs/proxy/call_hooks - LiteLLM production docs: https://docs.litellm.ai/docs/proxy/prod - LiteLLM prompt compression docs: https://docs.litellm.ai/docs/completion/prompt_compression

What Headroom Does

Headroom compresses large prompt/context payloads before they reach the LLM provider. The useful cases are long tool outputs, logs, file contents, RAG chunks, API responses, and long conversation histories.

Headroom advertises: - Library mode: compress(messages) - LiteLLM callback: litellm.callbacks = [HeadroomCallback()] - ASGI middleware: app.add_middleware(CompressionMiddleware) - Standalone proxy: headroom proxy - MCP integration and agent wrappers

For LiteLLM proxy specifically, the official Headroom docs show two supported integration paths:

litellm_settings:
  callbacks: ["headroom.integrations.litellm_callback.HeadroomCallback"]

or ASGI middleware:

from litellm.proxy.proxy_server import app
from headroom.integrations.asgi import CompressionMiddleware

app.add_middleware(CompressionMiddleware)

The callback compresses messages in LiteLLM’s pre_call_hook before LiteLLM sends the request to the provider. LiteLLM’s own docs confirm async_pre_call_hook is the correct hook for modifying incoming request data before the LLM API call.

Integration Options

Option A: LiteLLM Callback

Recommended for the current EKS setup.

Pros: - Minimal change: Docker dependency plus config.yaml - No new Service, Deployment, Ingress, or sidecar - Works inside the existing LiteLLM request lifecycle - Lower operational risk - Good match for single replica

Cons: - Does not expose Headroom’s ASGI response headers such as x-headroom-tokens-saved - Savings visibility is mainly through logs or provider/LiteLLM spend deltas unless you add custom logging - Constructor tuning via YAML is limited; use a custom callback module if you need parameters

Option B: ASGI Middleware

Good second phase if you want response headers or endpoint-level interception.

Pros: - Intercepts /v1/messages, /v1/chat/completions, /v1/responses, and /chat/completions - Adds response headers such as x-headroom-compressed, x-headroom-tokens-before, x-headroom-tokens-after, and x-headroom-tokens-saved

Cons: - Requires changing the app startup path, because the official LiteLLM image normally starts through the litellm CLI - More fragile across LiteLLM image upgrades

Option C: Separate Headroom Proxy

Not recommended for this first pass.

Pros: - Zero changes to LiteLLM internals if clients can point to Headroom first - Can enable proxy features such as --mode token or --mode cache

Cons: - Adds another hop: client -> Headroom -> LiteLLM -> provider - Needs extra auth/networking decisions - More moving pieces on EKS - Harder to reason about failures and request IDs

Target Architecture

Client / Open WebUI / internal apps
  -> Kubernetes Service / Ingress
  -> LiteLLM pod, single replica
     -> HeadroomCallback inside LiteLLM process
     -> provider SDK/API via LiteLLM
  -> Bedrock / OpenAI / Anthropic / Azure / etc.

Prerequisites

  • Existing LiteLLM proxy config mounted into the pod, usually config.yaml
  • Existing mirrored official LiteLLM image build pipeline
  • Python 3.10+ in the base image, required by Headroom
  • ECR repository for the custom image
  • EKS deployment or Helm values under your control
  • Baseline metrics for comparison:
  • LiteLLM request count
  • LiteLLM spend tracking, if enabled
  • provider-side input/output token counts
  • p50/p95 latency
  • error rate

Step 1: Pin Versions

Do not use unpinned latest for the production image. Pin both:

  • LiteLLM base image tag
  • headroom-ai package version

As of this research, PyPI lists headroom-ai 0.28.0, released 2026-06-29, requiring Python >=3.10.

Example:

ARG LITELLM_IMAGE=docker.litellm.ai/berriai/litellm:<pin-this>
FROM ${LITELLM_IMAGE}

USER root
RUN pip install --no-cache-dir "headroom-ai==0.28.0"
USER litellm

If the base image does not use litellm as the final user, inspect it and preserve the original user instead of blindly setting USER litellm.

Step 2: Enable Callback in LiteLLM Config

Add the Headroom callback to the existing LiteLLM config.

litellm_settings:
  callbacks:
    - headroom.integrations.litellm_callback.HeadroomCallback

If you already have callbacks, append Headroom instead of replacing them.

Example:

litellm_settings:
  callbacks:
    - langfuse
    - headroom.integrations.litellm_callback.HeadroomCallback

Callback order matters if another callback mutates messages. Put Headroom after policy/guardrail callbacks that need raw input, but before observability callbacks if you want logs to reflect compressed provider-bound prompts. If observability must capture original prompts, put observability before Headroom or keep raw prompt capture in the app layer.

Step 3: Add Environment Variables

For OSS local mode, do not set HEADROOM_API_KEY. The callback uses local compression by default. If HEADROOM_API_KEY is present, Headroom switches to cloud compression according to its integration source.

Recommended env:

env:
  - name: HEADROOM_TELEMETRY
    value: "off"
  - name: HEADROOM_STATELESS
    value: "true"

Notes: - HEADROOM_TELEMETRY defaults to opt-in/off in the docs, but set it explicitly for enterprise hygiene. - HEADROOM_STATELESS=true reduces filesystem writes. Use this for the first rollout unless you deliberately want local memory/persistent savings. - If you later use persistent Headroom features, mount a PVC and set HEADROOM_WORKSPACE_DIR.

Do not set these unless you intentionally want cloud mode:

HEADROOM_API_KEY
HEADROOM_API_URL

Step 4: Build and Push to ECR

Example:

AWS_ACCOUNT_ID="$(aws sts get-caller-identity --query Account --output text)"
AWS_REGION="ap-southeast-1"
ECR_REPO="litellm-headroom"
IMAGE_TAG="litellm-<version>-headroom-0.28.0"

aws ecr get-login-password --region "$AWS_REGION" \
  | docker login --username AWS --password-stdin "$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com"

docker build \
  --build-arg LITELLM_IMAGE=docker.litellm.ai/berriai/litellm:<pin-this> \
  -t "$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/$ECR_REPO:$IMAGE_TAG" .

docker push "$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/$ECR_REPO:$IMAGE_TAG"

Step 5: Update EKS Deployment or Helm Values

If using plain Deployment:

spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: litellm
          image: <account>.dkr.ecr.<region>.amazonaws.com/litellm-headroom:litellm-<version>-headroom-0.28.0
          env:
            - name: HEADROOM_TELEMETRY
              value: "off"
            - name: HEADROOM_STATELESS
              value: "true"
          readinessProbe:
            httpGet:
              path: /health/readiness
              port: 4000
          livenessProbe:
            httpGet:
              path: /health/liveliness
              port: 4000

LiteLLM docs list /health, /health/readiness, and /health/liveliness as proxy health endpoints.

LiteLLM production docs recommend one Uvicorn worker per pod by default and resource sizing roughly 1 CPU and 4Gi memory per worker. For your single replica, start with the existing sizing if it is stable, then watch CPU and latency after enabling compression.

Step 6: Smoke Test

Port-forward:

kubectl -n <namespace> port-forward deploy/<litellm-deployment> 4000:4000

Send a deliberately large prompt where compression should trigger:

curl -sS http://localhost:4000/chat/completions \
  -H "Authorization: Bearer $LITELLM_MASTER_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "<your-litellm-model-name>",
    "messages": [
      {
        "role": "user",
        "content": "Summarize this log and identify the root cause:\n<large repeated log payload here>"
      }
    ]
  }'

Check pod logs:

kubectl -n <namespace> logs deploy/<litellm-deployment> --tail=200 | grep -i headroom

Expected signal:

Headroom: <before>→<after> tokens (saved <n>, <percent>%)

If there are no Headroom logs: - Confirm headroom-ai is installed in the image. - Confirm the callback exists in the effective LiteLLM config. - Confirm the pod restarted onto the new image. - Confirm traffic is hitting /chat/completions or compatible completion routes.

Step 7: Measure Savings

Use a before/after window, not a single request.

Minimum useful measurement:

  1. Pick a representative workload window, such as one business day or a replay set.
  2. Record baseline:
  3. request count
  4. provider input tokens
  5. provider output tokens
  6. cost
  7. p50/p95 latency
  8. error rate
  9. Enable Headroom for the same traffic type.
  10. Compare:
  11. input token reduction
  12. output token change
  13. total cost delta
  14. latency delta
  15. answer quality issues

Important: Headroom mainly reduces input/context tokens unless you use proxy-level output shaping. LiteLLM callback mode should be evaluated primarily on provider input token reduction.

Rollout Plan

Phase 0: Local Image Validation

Run the custom image locally with the production-ish config and send one synthetic long request.

Exit criteria: - LiteLLM starts - Headroom imports successfully - Normal short request still works - Long request logs savings

Phase 1: EKS Shadow/Canary

Because you currently run one replica, the safest first EKS validation is either: - deploy a second temporary LiteLLM service with the Headroom image and point only test clients to it, or - deploy during a low-traffic window and be ready to roll back the image.

Exit criteria: - no startup crash - no auth/config regression - no increase in 5xx - p95 latency acceptable - observed savings on long prompts

Phase 2: Production Enablement

Roll the main Deployment to the new image and config.

Watch for 30-60 minutes: - pod restarts - LiteLLM errors - provider errors - p95/p99 latency - “Headroom compression failed, using original messages” warnings - token/cost trend

Phase 3: Tune or Expand

If savings are material and stable, decide whether to: - stay on callback mode - add custom Headroom callback instantiation for stricter tuning - move to ASGI middleware for response headers - evaluate Headroom Cloud/self-hosted team deployment only if governance accepts it

Rollback

Fast rollback:

kubectl -n <namespace> rollout undo deploy/<litellm-deployment>

or set the old image tag:

kubectl -n <namespace> set image deploy/<litellm-deployment> \
  litellm=<old-ecr-image-tag>

Config rollback: - Remove headroom.integrations.litellm_callback.HeadroomCallback from litellm_settings.callbacks. - Restart the pod.

Because the callback fails open according to the source code path, compression failures should log a warning and use original messages. Still, do not rely on fail-open behavior as your only rollback.

Risks and Mitigations

Risk: Data Governance

Local callback mode keeps compression in-process. Cloud mode sends messages to Headroom Cloud if HEADROOM_API_KEY is set. Do not set HEADROOM_API_KEY unless cloud processing has been reviewed.

Risk: Latency Overhead

Compression adds CPU work before provider calls. Monitor p95/p99 latency and CPU. If traffic grows, prefer scaling pods over increasing workers inside the same pod, matching LiteLLM’s production guidance of one Uvicorn worker per pod.

Risk: Quality Regression

Compression can alter the prompt context. Use representative regression prompts, especially for: - tool-heavy agent traces - RAG answers requiring exact citations - compliance-sensitive responses - prompts where exact logs or JSON fields matter

Risk: Observability Confusion

LiteLLM/provider token counts will show the compressed prompt, not necessarily the original logical payload. Decide whether app-layer observability should retain original prompt size for internal ROI tracking.

Risk: Single Replica Restart

With one replica, image rollout causes downtime unless your deployment has enough surge capacity and readiness behavior. Use low-traffic rollout or temporarily set replicas: 2 for the deployment window if your DB/Redis/config supports it.

Risk: Callback Limit or Callback Order

LiteLLM has callback limits and every callback adds per-request processing. If you already use many callbacks, check callback count and latency.

Recommended First Implementation

Use local callback mode:

FROM docker.litellm.ai/berriai/litellm:<pin-this>

USER root
RUN pip install --no-cache-dir "headroom-ai==0.28.0"
USER litellm

Config:

litellm_settings:
  callbacks:
    - headroom.integrations.litellm_callback.HeadroomCallback

Env:

- name: HEADROOM_TELEMETRY
  value: "off"
- name: HEADROOM_STATELESS
  value: "true"

This should be the first POC because it is the smallest change with the cleanest rollback.
